# 작업 실행 계획 가이드

이 문서는 **개별 Task의 구체적인 실행 계획을 작성하기 위한 가이드**이다.

---

## 1. 작업 실행 계획의 목적

작업 실행 계획은 다음을 위해 작성한다:

- 리스크 조기 발견: 착수 전 복잡도 파악
- 구현 품질 향상: 설계 후 코딩
- 명확한 범위: Scope Creep 방지
- 의사결정 기록: 왜 이렇게 했는지
- 팀 커뮤니케이션: 피드백 수렴

실행 계획은 **"어떻게 만들 것인가"를 구체화**하는 과정이다.

---

## 2. 언제 작성하는가

### 필수 (Must)

- ✅ 새로운 기술 사용 시
- ✅ 복잡한 Task (4시간 이상)
- ✅ 리스크 높은 Task
- ✅ 팀 협업 필요 시

### 선택 (Optional)

- ⚪ 간단한 Task (2-3시간)
- ⚪ 반복적 작업
- ⚪ 익숙한 기술

### 불필요 (Skip)

- ❌ 매우 간단 (1시간 이하)
- ❌ 설정 변경, 텍스트 수정

---

## 3. 작성 시점

```
Task Breakdown 완료
  ↓
Task 선택 (오늘 할 일)
  ↓
실행 계획 작성 (30분) ← 여기
  ↓
계획 리뷰 (15분)
  ↓
구현 시작
```

**타이밍:**
- Task 착수 직전 (당일 아침)
- Task 시간의 10% (6h Task → 30분 계획)

---

## 4. 실행 계획 vs Task Breakdown

| 구분 | Task Breakdown | Execution Plan |
|------|----------------|----------------|
| 시점 | 구현 전 (weeks) | 착수 직전 (hours) |
| 대상 | Phase 전체 | 1개 Task |
| 상세도 | 4-8시간 단위 | 30분-1시간 단위 |
| 목적 | 전체 계획 | 구체적 구현 |
| 작성자 | Tech Lead | 담당 개발자 |

**예시:**

Task Breakdown:
```
Task 1.5: PDF 파서 (6h)
- pypdf로 텍스트 추출
- 에러 핸들링
```

Execution Plan:
```
1. 조사 (1h): pypdf vs pdfplumber → pypdf 선택
2. 기본 구현 (2h): PDFParser 클래스
3. 메타데이터 (1h): 페이지 번호, 파일명
4. 에러 처리 (1h): 3가지 에러 타입
5. 테스트 (1h): 5개 케이스
```

---

## 5. 작성 원칙

### 5.1 적절한 상세도

```
✅ 좋은 예 (30분-1시간 단위):
Step 1: 조사 및 설계 (1h)
Step 2: 기본 구현 (2h)
Step 3: 에러 처리 (1h)
Step 4: 테스트 (1h)

❌ 너무 상세 (10-15분 단위):
Step 1.1: 파일 열기 (10분)
Step 1.2: 첫 페이지 읽기 (5분)
...

❌ 너무 추상 (시간 없음):
Step 1: 구현
Step 2: 테스트
```

### 5.2 시간 배분

```
일반적 비율:
- 조사/설계: 15-20%
- 구현: 60-70%
- 테스트: 15-20%

예시 (6시간 Task):
- 조사: 1h (17%)
- 구현: 4h (67%)
- 테스트: 1h (17%)
```

### 5.3 테스트 케이스 명시

```
✅ 구체적으로:
테스트 계획:
- test_valid_pdf_parsing()
- test_corrupted_pdf_error()
- test_encrypted_pdf_warning()
- test_multipage_extraction()
- test_metadata_accuracy()

❌ 추상적으로:
테스트 계획:
- 단위 테스트
```

---

## 6. 구성 요소

### 필수 항목

```
1. Task 정보
   - Task ID, Name
   - 예상 시간
   - 담당자
   
2. 조사 및 준비
   - 기술 조사
   - 설계 결정
   
3. 구현 단계
   - Step별 작업
   - 각 30분-1시간
   
4. 테스트 계획
   - 테스트 케이스 목록
   
5. 리스크
   - 예상 어려움
   - 대응 방안
   
6. Definition of Done
   - 완료 체크리스트
```

### 선택 항목

```
7. 참고 자료
   - 문서 링크
   - 코드 예시
   
8. 의사결정 기록
   - 선택지와 이유
   
9. 후속 작업
   - 다음 Task 연결
```

---

## 7. 작성 프로세스

### Step 1: Task 정보 확인

```
Task Breakdown 읽기:
- Task 1.5 내용
- Acceptance Criteria
- Technical Details
```

### Step 2: 조사 및 설계

```
필요 시 조사:
- 라이브러리 비교
- API 문서 확인
- 유사 코드 검색

설계 결정:
- 클래스 구조
- 함수 시그니처
- 에러 처리 전략
```

### Step 3: 단계별 분해

```
Task를 30분-1시간 단위로:
- 각 Step이 검증 가능
- 명확한 산출물
- 독립적 테스트
```

### Step 4: 테스트 계획

```
테스트 케이스 나열:
- 정상 케이스
- 에러 케이스
- 경계 케이스
```

### Step 5: 리스크 파악

```
예상 어려움:
- 기술적 블로커
- 외부 의존성
- 시간 초과 가능성

대응 방안:
- 대안 기술
- 일정 조정
- 도움 요청
```

---

## 8. 리뷰 프로세스

### Peer Review (15분)

```
공유:
- 실행 계획 문서
- Slack/Teams에 공유

피드백:
- "이 라이브러리가 더 나을 것 같아요"
- "이전 프로젝트에서 이런 문제 있었어요"
- "테스트 케이스에 X 추가하세요"

반영:
- 계획 수정
- 리스크 업데이트
```

### Lead Approval

```
승인 기준:
- [ ] 구현 방법 적절
- [ ] 시간 추정 합리적
- [ ] 리스크 파악됨
- [ ] 테스트 충분

승인 후:
- 구현 시작
```

---

## 9. 실행 중 추적

### 진행 상황 기록

```
계획:
Step 1: 조사 (1h)
Step 2: 구현 (2h)
Step 3: 테스트 (1h)

실제:
Step 1: ✅ 0.5h (빨리 끝남)
Step 2: 🔄 2.5h (진행 중, 예상 초과)
Step 3: ⏳ 예정

→ Step 2에서 지연 발견
→ 다음 Task 일정 조정
```

### 계획 vs 실제

```
예상: 6h
실제: 6.5h

차이 분석:
- Step 2가 복잡했음
- 라이브러리 API 변경
- 다음번 유사 Task는 +30분
```

## 10. 템플릿 사용

작업 실행 계획 시 `TEMPLATE.md`를 사용한다:

- Task 정보부터 순서대로
- 모든 필수 항목 채우기
- 선택 항목은 필요 시만

---

## 11. 모범 사례

### DO ✅

- Task 착수 직전 작성 (당일 아침)
- 30분 이내 작성 (빠르게)
- Peer 리뷰 받기
- 테스트 케이스 구체적으로
- 리스크 솔직하게

### DON'T ❌

- 완벽한 계획 추구 (Analysis Paralysis)
- 너무 상세하게 (10분 단위)
- 혼자 결정 (피드백 없이)
- 계획만 하고 실행 안 함
- 계획 무시하고 즉흥적 구현

---

## 12. 효과

### 시간 효율

```
계획 시간: +30분
구현 시간: -3.5시간
순 절감: 3시간 (50%)
```

### 품질 향상

```
계획 없이: 60점
계획 후: 90점
개선: +30점 (50%)
```

### 리스크 감소

```
일정 지연: 70% → 20%
재작업: 50% → 10%
```

---

## 13. 최종 원칙

작업 실행 계획은 **"삽질을 줄이기 위한 투자"**이다.

- 30분 투자 → 3시간 절약
- 착수 전 설계 → 일관성 있는 코드
- 리스크 조기 발견 → 블로커 방지
- 의사결정 기록 → 컨텍스트 유지

**"Measure twice, cut once"** - 두 번 재고, 한 번 자른다.
