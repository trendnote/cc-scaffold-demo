# 기술 스택 결정 가이드

이 문서는 PRD를 기반으로
**프로젝트에 적합한 기술 스택을 선택하기 위한 가이드**이다.

---

## 1. 기술 스택 결정의 목적

기술 스택 결정은 다음을 명확히 하기 위해 수행한다.

- 어떤 기술을 사용할 것인가
- 왜 그 기술을 선택했는가
- 해당 기술이 요구사항을 충족하는가
- 팀이 해당 기술을 습득/운영할 수 있는가

기술 스택은 **구현 방식이 아니라 도구 선택**이다.
아키텍처 설계는 별도 문서로 작성한다.

---

## 2. 기술 스택 결정 시점

다음 상황에서 기술 스택을 결정한다.

- PRD가 승인된 직후
- 새로운 프로젝트 시작
- 기존 프로젝트의 주요 기술 변경
- 기술 부채 해소를 위한 재설계

---

## 3. 기술 스택 결정 원칙

### 3.1 요구사항 우선
- PRD의 요구사항을 모두 충족해야 함
- 성능, 보안, 확장성 등 비기능 요구사항 고려
- Out of Scope는 고려하지 않음

### 3.2 팀 역량 고려
- 팀이 학습 가능한 기술인가
- 유지보수 가능한가
- 채용이 가능한가

### 3.3 실용성 우선
- 최신 기술 > 검증된 기술 (검증된 것 우선)
- 복잡한 기술 > 단순한 기술 (단순한 것 우선)
- 유행 > 실용성 (실용성 우선)

### 3.4 생태계 고려
- 커뮤니티가 활발한가
- 문서가 풍부한가
- 라이브러리/도구가 충분한가
- 장기 지원이 보장되는가

---

## 4. 기술 스택 카테고리

기술 스택은 다음 카테고리로 구성된다.

### 4.1 Backend
- 프로그래밍 언어
- 웹 프레임워크
- API 스타일 (REST, GraphQL 등)

### 4.2 Frontend
- 프레임워크 (React, Vue, Svelte 등)
- 언어 (TypeScript, JavaScript)
- 스타일링 (CSS, Tailwind 등)

### 4.3 Database
- RDBMS (PostgreSQL, MySQL 등)
- NoSQL (MongoDB, Redis 등)
- 특수 DB (Vector DB, Graph DB 등)

### 4.4 Infrastructure
- 클라우드 (AWS, GCP, Azure, 온프레미스)
- 컨테이너 (Docker, Kubernetes)
- CI/CD (GitHub Actions, Jenkins 등)

### 4.5 External Services
- 인증 (Auth0, Clerk, 자체 구현)
- 결제 (Stripe, Toss Payments)
- 모니터링 (Datadog, New Relic)
- 기타 Third-party API

---

## 5. 선택 기준

각 기술을 평가할 때 다음 기준을 사용한다.

### 5.1 필수 기준 (Must-Have)
- PRD 요구사항 충족
- 성능 목표 달성 가능
- 보안 요구사항 만족
- 라이선스 적합성

### 5.2 중요 기준 (Should-Have)
- 팀의 기존 경험
- 학습 곡선
- 커뮤니티 크기
- 문서 품질

### 5.3 선택 기준 (Nice-to-Have)
- 최신 버전
- 유행
- 개인 선호도

---

## 6. 기술 스택과 다른 문서의 관계

```
PRD (요구사항)
  ↓
Tech Stack (도구 선택) ← 이 문서
  ↓
Architecture (구조 설계)
  ↓
Implementation (구현)
```

- PRD 변경 → Tech Stack 재검토 필요
- Tech Stack 변경 → Architecture 재설계 필요

---

## 7. 템플릿 사용

기술 스택 결정 시 `TEMPLATE.md`를 기준으로 작성한다.

- 모든 섹션을 채울 필요는 없음
- 비워둔 경우, 그 이유를 명시

---

## 8. 최종 원칙

기술 스택 결정은 "최신 기술을 쓰기 위한 결정"이 아니라
**요구사항을 가장 잘 충족하는 도구를 선택하는 과정**이다.
