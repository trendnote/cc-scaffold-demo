# Product Requirements Document (PRD) 작성 가이드

이 문서는 카카오페이 프로젝트에서
**요구사항을 명확히 정의하기 위한 PRD 작성 가이드**이다.

Claude Code를 활용한 개발에서는
PRD가 **개발의 출발점이자, 모든 판단의 기준**이 된다.

---

## 1. PRD의 목적

PRD는 다음을 명확히 하기 위해 작성한다.

- 무엇을 만들 것인가
- 왜 만들어야 하는가
- 무엇을 만들지 말아야 하는가
- 성공 기준은 무엇인가

PRD는 **설계 문서도 아니고, 구현 문서도 아니다.**
비즈니스 요구사항과 의도를 정확히 전달하는 문서다.

---

## 2. PRD는 언제 작성해야 하나

다음 중 하나라도 해당되면 PRD 작성이 필요하다.

- 새로운 기능 또는 서비스 개발
- 기존 기능의 의미 있는 변경
- 사용자 경험(UX) 변화
- 정책/규칙/조건 변경
- 여러 팀이 함께 작업하는 경우

---

## 3. PRD 작성 원칙

### 3.1 명확성 우선
- 애매한 표현을 피한다
- “적당히”, “유연하게”, “가능하면” 같은 표현은 지양한다
- 해석의 여지가 있다면 구체화한다

---

### 3.2 해결책보다 문제 정의
- “어떻게 만들 것인가”보다
  “어떤 문제가 있는가”에 집중한다
- 기술 구현 방식은 PRD의 범위가 아니다

---

### 3.3 범위의 명시
- 반드시 다음을 포함한다:
  - In Scope (이번에 포함되는 것)
  - Out of Scope (이번에 포함되지 않는 것)
- Out of Scope는 매우 중요하다

---

### 3.4 성공 기준의 명문화
- “잘 되었는지”를 판단할 수 있어야 한다
- 가능하면 정량 지표를 포함한다
- 최소 성공 조건(Minimum Success Criteria)을 정의한다

---

## 4. Claude Code와 PRD

Claude Code는 PRD를 기반으로 다음을 수행한다.

- 기능 범위 해석
- 엣지 케이스 추론
- 테스트 케이스 도출
- 설계 대안 제시

PRD가 불충분하면
Claude Code는 반드시 질문을 통해 보완을 요청해야 한다.

---

## 5. 권장 PRD 템플릿

PRD 작성 시 `TEMPLATE.md`를 기준으로 작성한다.

- 모든 섹션을 반드시 채울 필요는 없다
- 다만 비워둔 경우, 그 이유를 명시한다

---

## 6. PRD와 개발 산출물의 관계

- PRD → 설계 → 코드 → 테스트 → 문서
- PRD 변경은 곧 요구사항 변경이다
- PRD 변경 시, 관련 코드와 문서도 함께 검토한다

---

## 7. 최종 원칙
PRD는 “문서를 남기기 위한 문서”가 아니라 **잘못된 개발을 예방하기 위한 도구**다.
